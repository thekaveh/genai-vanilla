# Plan A — Cloudflared Tunnel Service Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a `cloudflared` service (Cloudflare Tunnel sidecar) that connects outbound to Cloudflare and proxies inbound HTTPS traffic to the existing Kong gateway, giving the stack TLS + a public domain with zero open inbound ports.

**Architecture:** A new single-container service `services/cloudflared/` following the repo's standard manifest+fragment pattern. It runs `cloudflare/cloudflared` with a `TUNNEL_TOKEN`, joins `backend-network`, and reaches Kong by service DNS (`kong-api-gateway:8000`). It is egress-only: `data_flow.calls: [kong]`; Kong needs no route for it. Default `CLOUDFLARED_SOURCE=disabled` so nothing changes for existing users until they opt in.

**Tech Stack:** Docker Compose fragment + YAML manifest; bootstrapper manifest loader / topology / env-assembler / docs-regen; pytest.

## Global Constraints

- `main` is protected — land via PR with the 3 `services-lint` checks green; never push to `main`.
- Commit messages: terse third-person verb, no emoji, no Claude `Co-Authored-By` trailer.
- `.env.example` is GENERATED from manifests by `services/env_assembler.py`; never hand-edit it — change the manifest and regenerate.
- Port defaults are computed by `services/topology.py`; manifest `*_PORT` env vars carry NO `default:`.
- Service docs (`README.md` + `architecture.svg/.html`) are regenerated by `bootstrapper.docs.regen`; a drift gate enforces they match.
- The compose byte-equivalence baseline is `bootstrapper/tests/fixtures/rendered_config_baseline.yml`; adding a `disabled`-by-default service that emits a `replicas: 0` container changes rendered config — regen the baseline if `test_fragment_equivalence` fails (note: local Docker Compose version can differ from CI; if local regen produces large drift, regenerate the baseline via the CI artifact path documented in the repo).
- Init-container / image conventions: vanilla `alpine` + inline `apk`, never a Dockerfile that clobbers a global tag. (cloudflared uses the upstream `cloudflare/cloudflared` image directly — no init container needed.)

- [ ] **Step 1: Write the manifest**

```yaml
# services/cloudflared/service.yml
name: cloudflared
label: "Cloudflare Tunnel (public edge)"
category: infra
docs: services/cloudflared/README.md

containers:
  - cloudflared

images:
  - var: CLOUDFLARED_IMAGE
    default: "cloudflare/cloudflared:2025.8.1"
    container: cloudflared
    notes: "Cloudflare Tunnel daemon. Pin a dated tag; bump deliberately."

sources:
  cloudflared:
    var: CLOUDFLARED_SOURCE
    default: disabled
    options:
      - id: container
        description: "Run the tunnel daemon as a container (proxies to Kong)."
      - id: disabled
        description: "No public edge (default)."

env:
  - name: CLOUDFLARED_PORT
    # default removed — computed by services/topology.py slot allocator
    description: "Host port for cloudflared local metrics (in-container 2000). Bound to localhost in prod."
  - name: CLOUDFLARE_TUNNEL_TOKEN
    default: ""
    secret: true
    description: |
      Cloudflare named-tunnel token (Zero Trust > Networks > Tunnels). When
      CLOUDFLARED_SOURCE=container this must be set or the daemon exits. The
      tunnel's public hostname is configured in the Cloudflare dashboard to
      point at http://kong-api-gateway:8000.
  - name: CLOUDFLARED_SCALE
    auto_managed: true
    description: "Set to 1 when CLOUDFLARED_SOURCE=container, 0 when disabled."

depends_on:
  required:
    - kong
  optional: []

rows:
  - display_name: "Cloudflare Tunnel"
    source_var: CLOUDFLARED_SOURCE
    port_var: CLOUDFLARED_PORT
    description: "Outbound tunnel to Cloudflare; terminates TLS at the edge and proxies to Kong."

exports: []

runtime_sc:
  cloudflared:
    container:
      scale: 1
      environment:
        TUNNEL_TOKEN: ${CLOUDFLARE_TUNNEL_TOKEN}
        TUNNEL_METRICS: "0.0.0.0:2000"
      deploy: {}
      extra_hosts: []
    disabled:
      scale: 0
      environment: {}
      deploy: {}
      extra_hosts: []

data_flow:
  calls:
    - kong
```

### Task 2: Compose fragment (`services/cloudflared/compose.yml`)

**Files:**
- Create: `services/cloudflared/compose.yml`
- Modify: `docker-compose.yml` (add include line)
- Test: `bootstrapper/tests/test_fragment_equivalence.py`

**Interfaces:**
- Consumes: `CLOUDFLARED_IMAGE`, `CLOUDFLARED_SCALE`, `CLOUDFLARE_TUNNEL_TOKEN`, `PROJECT_NAME` from env; `backend-network`; `kong-api-gateway` service (Kong's compose service id).

- [ ] **Step 1: Write the fragment**

```yaml
# Cloudflare Tunnel — outbound edge that terminates TLS at Cloudflare and
# proxies to Kong. Egress-only: no published ingress port, no Kong route.
# Disabled by default (CLOUDFLARED_SCALE=0).

services:
  cloudflared:
    image: ${CLOUDFLARED_IMAGE}
    container_name: ${PROJECT_NAME}-cloudflared
    restart: unless-stopped
    deploy:
      replicas: ${CLOUDFLARED_SCALE:-0}
    command: ["tunnel", "--no-autoupdate", "run"]
    environment:
      TUNNEL_TOKEN: ${CLOUDFLARE_TUNNEL_TOKEN}
      TUNNEL_METRICS: "0.0.0.0:2000"
    healthcheck:
      test: ["CMD", "cloudflared", "tunnel", "--metrics", "0.0.0.0:2000", "ready"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 20s
    depends_on:
      kong-api-gateway:
        condition: service_healthy
    networks:
      - backend-network
```

Note: the tunnel daemon is configured entirely by `TUNNEL_TOKEN` + the Cloudflare dashboard (public hostname → `http://kong-api-gateway:8000`). No host port is published (it dials out). The metrics port (2000) stays in-container; expose it only if you later wire it to Prometheus.
